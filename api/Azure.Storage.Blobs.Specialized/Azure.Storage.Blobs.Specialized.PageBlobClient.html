    Class PageBlobClient

    The PageBlobClient allows you to manipulate Azure Storage page blobs.

    Page blobs are a collection of 512-byte pages optimized for random read and write operations. To create a page blob, you initialize the page blob and specify the maximum size the page blob will grow. To add or update the contents of a page blob, you write a page or pages by specifying an offset and a range that align to 512-byte page boundaries. A write to a page blob can overwrite just one page, some pages, or up to 4 MB of the page blob. Writes to page blobs happen in-place and are immediately committed to the blob. The maximum size for a page blob is 8 TB.

    StartCopyIncremental(Uri, String, Nullable<PageBlobAccessConditions>, CancellationToken)

    The StartCopyIncremental(Uri, String, Nullable<PageBlobAccessConditions>, CancellationToken) operation starts copying a snapshot of the sourceUri page blob to this page blob. The snapshot is copied such that only the differential changes between the previously copied snapshot are transferred to the destination. The copied snapshots are complete copies of the original snapshot and can be read or copied from as usual. You can check the CopyStatus returned from the GetProperties(Nullable<BlobAccessConditions>, CancellationToken) to determine if the copy has completed.

    Remarks

    A StorageRequestFailedException will be thrown if a failure occurs.

    The destination of an incremental copy must either not exist, or must have been created with a previous incremental copy from the same source blob. Once created, the destination blob is permanently associated with the source and may only be used for incremental copies.

    The GetProperties(Nullable<BlobAccessConditions>, CancellationToken), GetBlobs(Nullable<GetBlobsOptions>, CancellationToken), and GetBlobsByHierarchy(String, Nullable<GetBlobsOptions>, CancellationToken) operations indicate whether the blob is an incremental copy blob created in this way. Incremental copy blobs may not be downloaded directly. The only supported operations are GetProperties(Nullable<BlobAccessConditions>, CancellationToken), StartCopyIncremental(Uri, String, Nullable<PageBlobAccessConditions>, CancellationToken), and Delete(Nullable<DeleteSnapshotsOption>, Nullable<BlobAccessConditions>, CancellationToken). The copied snapshots may be read and deleted as usual.

    An incremental copy is performed asynchronously on the service and must be polled for completion. You can poll GetProperties(Nullable<BlobAccessConditions>, CancellationToken) and check CopyStatus to determine when the copy has completed. When the copy completes, the destination blob will contain a new snapshot. The GetProperties(Nullable<BlobAccessConditions>, CancellationToken) operation returns the snapshot time of the newly created snapshot.

    The first time an incremental copy is performed on a destination blob, a new blob is created with a snapshot that is fully copied from the source. Each subsequent call to StartCopyIncremental(Uri, String, Nullable<PageBlobAccessConditions>, CancellationToken) will create a new snapshot by copying only the differential changes from the previously copied snapshot. The differential changes are computed on the server by issuing a GetPageRanges(Nullable<HttpRange>, String, Nullable<PageBlobAccessConditions>, CancellationToken) call on the source blob snapshot with prevSnapshot set to the most recently copied snapshot. Therefore, the same restrictions on GetPageRanges(Nullable<HttpRange>, String, Nullable<PageBlobAccessConditions>, CancellationToken) apply to StartCopyIncremental(Uri, String, Nullable<PageBlobAccessConditions>, CancellationToken). Specifically, snapshots must be copied in ascending order and if the source blob is recreated using UploadPages(Stream, Int64, Byte[], Nullable<PageBlobAccessConditions>, IProgress<StorageProgress>, CancellationToken) or StartCopyFromUri(Uri, IDictionary<String,String>, Nullable<AccessTier>, Nullable<BlobAccessConditions>, Nullable<BlobAccessConditions>, Nullable<RehydratePriority>, CancellationToken) then StartCopyIncremental(Uri, String, Nullable<PageBlobAccessConditions>, CancellationToken) on new snapshots will fail.

    The additional storage space consumed by the copied snapshot is the size of the differential data transferred during the copy. This can be determined by performing a GetPageRangesDiff(Nullable<HttpRange>, String, String, Nullable<PageBlobAccessConditions>, CancellationToken) call on the snapshot to compare it to the previous snapshot.

    A StorageRequestFailedException will be thrown if a failure occurs.

    The destination of an incremental copy must either not exist, or must have been created with a previous incremental copy from the same source blob. Once created, the destination blob is permanently associated with the source and may only be used for incremental copies.

    The GetPropertiesAsync(Nullable<BlobAccessConditions>, CancellationToken), GetBlobsAsync(Nullable<GetBlobsOptions>, CancellationToken), and GetBlobsByHierarchyAsync(String, Nullable<GetBlobsOptions>, CancellationToken) operations indicate whether the blob is an incremental copy blob created in this way. Incremental copy blobs may not be downloaded directly. The only supported operations are GetPropertiesAsync(Nullable<BlobAccessConditions>, CancellationToken), StartCopyIncrementalAsync(Uri, String, Nullable<PageBlobAccessConditions>, CancellationToken), and DeleteAsync(Nullable<DeleteSnapshotsOption>, Nullable<BlobAccessConditions>, CancellationToken). The copied snapshots may be read and deleted as usual.

    An incremental copy is performed asynchronously on the service and must be polled for completion. You can poll GetPropertiesAsync(Nullable<BlobAccessConditions>, CancellationToken) and check CopyStatus to determine when the copy has completed. When the copy completes, the destination blob will contain a new snapshot. The GetPropertiesAsync(Nullable<BlobAccessConditions>, CancellationToken) operation returns the snapshot time of the newly created snapshot.

    The first time an incremental copy is performed on a destination blob, a new blob is created with a snapshot that is fully copied from the source. Each subsequent call to StartCopyIncrementalAsync(Uri, String, Nullable<PageBlobAccessConditions>, CancellationToken) will create a new snapshot by copying only the differential changes from the previously copied snapshot. The differential changes are computed on the server by issuing a GetPageRangesAsync(Nullable<HttpRange>, String, Nullable<PageBlobAccessConditions>, CancellationToken) call on the source blob snapshot with prevSnapshot set to the most recently copied snapshot. Therefore, the same restrictions on GetPageRangesAsync(Nullable<HttpRange>, String, Nullable<PageBlobAccessConditions>, CancellationToken) apply to StartCopyIncrementalAsync(Uri, String, Nullable<PageBlobAccessConditions>, CancellationToken). Specifically, snapshots must be copied in ascending order and if the source blob is recreated using UploadPagesAsync(Stream, Int64, Byte[], Nullable<PageBlobAccessConditions>, IProgress<StorageProgress>, CancellationToken) or StartCopyFromUriAsync(Uri, IDictionary<String,String>, Nullable<AccessTier>, Nullable<BlobAccessConditions>, Nullable<BlobAccessConditions>, Nullable<RehydratePriority>, CancellationToken) then StartCopyIncrementalAsync(Uri, String, Nullable<PageBlobAccessConditions>, CancellationToken) on new snapshots will fail.

    The additional storage space consumed by the copied snapshot is the size of the differential data transferred during the copy. This can be determined by performing a GetPageRangesDiffAsync(Nullable<HttpRange>, String, String, Nullable<PageBlobAccessConditions>, CancellationToken) call on the snapshot to compare it to the previous snapshot.
